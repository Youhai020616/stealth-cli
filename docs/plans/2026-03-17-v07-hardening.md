## Task 2: 补齐剩余 5 个自定义错误 throw 点

**问题：** errors.js 定义了 7 个错误类，但只有 `BrowserLaunchError` 和 `NavigationError` 被 throw。剩余 5 个（`ExtractionError`、`TimeoutError`、`ProxyError`、`ProfileError`、`BlockedError`）是死代码。

**策略：** 在最合理的位置 throw 它们，让 `handleError` 的 `instanceof` 分支和 exit code 体系真正工作。

**Files:**
- Modify: `src/commands/search.js` — throw `BlockedError`
- Modify: `src/commands/extract.js` — throw `ExtractionError`
- Modify: `src/profiles.js` — throw `ProfileError`
- Modify: `src/proxy-pool.js` — throw `ProxyError`
- Modify: `src/browser.js` — throw `TimeoutError`（waitForReady 中可选）

- [ ] **Step 1: search.js — throw BlockedError**

search.js 第 85 行已有 Google 被拦截的检测逻辑，但用的是 `log.warn` 然后 return。改为 throw：

```js
// src/commands/search.js — import 区
import { handleError, BlockedError } from '../errors.js';

// 替换 Google blocked 检测（约第 85-95 行）:
if (googleExtractor.isBlocked(currentUrl)) {
  spinner.stop();
  throw new BlockedError('Google', currentUrl);
}
```

注意：原来的代码在 blocked 时如果 format=json 还会输出空结果。改为 throw 后由 handleError 统一处理，用户看到 hint "Try: --proxy, --warmup, --humanize"，exit code = 4。

- [ ] **Step 2: extract.js — 当选择器无结果时 throw ExtractionError**

extract.js 中如果 CSS 选择器匹配不到元素，当前静默返回空数组。加一个提示更友好：

```js
// src/commands/extract.js — import 区
import { handleError, ExtractionError } from '../errors.js';

// 在 result 计算完成后，console.log 之前（约第 140 行）：
if (result === null || result === undefined || (Array.isArray(result) && result.length === 0)) {
  if (opts.selector && opts.selector !== 'body') {
    throw new ExtractionError(
      `No elements found matching "${opts.selector}"`,
      { hint: `Check the CSS selector. Try: stealth browse ${url} -f snapshot` },
    );
  }
}
```

- [ ] **Step 3: profiles.js — throw ProfileError 替代普通 Error**

- [ ] **Step 4: proxy-pool.js — throw ProxyError**

proxy-pool.js 的 `testProxy` 在代理连接失败时应该 throw `ProxyError`：

```js
// src/proxy-pool.js — 在 testProxy 函数的 catch 块中
import { ProxyError } from './errors.js';

// testProxy 的返回结构不变（返回 { ok, error, ... }），
```

## Task 5: extract.js eval 字符串注入修复

**问题：** `extract.js` 的 CSS 选择器提取使用字符串模板拼接到 `page.evaluate` 中：

```js
result = await evalFn(`(() => {
  const elements = ${all}
    ? Array.from(document.querySelectorAll('${selector}'))
    : [document.querySelector('${selector}')].filter(Boolean);
  return elements.map(el => {
    if ('${attr || ''}') return el.getAttribute('${attr || ''}');
    return el.textContent?.trim() || '';
  });
})()`);
```

如果用户传入 `--selector "'); alert('xss"` 这样的值，会注入到 evaluate 字符串中。虽然 evaluate 运行在浏览器页面上下文中（不是 Node.js），但仍然是代码注入。

**修复：** 使用 `page.evaluate(fn, arg)` 的参数传递方式，而非字符串拼接。

**Files:**
- Modify: `src/commands/extract.js`

- [ ] **Step 1: 修改 extract.js 的 selector 分支**

将所有 `evalFn` 调用从字符串模板改为函数+参数：

```js
// 替换 extract.js 中 else 分支（CSS selector 提取，约第 130-140 行）
} else {
  const selector = opts.selector;
  const attr = opts.attr || null;
  const all = !!opts.all;

  result = await handle.page.evaluate(
    ({ selector, attr, all }) => {
      const elements = all
        ? Array.from(document.querySelectorAll(selector))
        : [document.querySelector(selector)].filter(Boolean);
      return elements.map(el => {
        if (attr) return el.getAttribute(attr);
        return el.textContent?.trim() || '';
      });
    },
    { selector, attr, all },
  );
}
```

同样，把其他分支（links、images、meta、headers）中的 `evalFn(string)` 也改为函数形式——虽然这些不接受用户输入，保持一致性更好。但改动面太大。先只改接受用户输入 `selector` 和 `attr` 的那个分支。
